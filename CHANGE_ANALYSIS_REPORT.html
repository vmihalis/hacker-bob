Main Advantages

Engineering Advantages

  • Clear ownership: auth, HTTP scan, storage, waves, findings, ranking, hunter brief, and coverage now have separate modules.
  • Safer future edits: a bug in wave merge no longer requires reading the entire server file.
  • Stable public API: export snapshot and dispatch tests protect compatibility.
  • Better test command: npm test now exercises MCP, prompt contracts, and write guards.

Product Advantages

  • Better hunter focus: briefs include real traffic, coverage, audit feedback, ranking, and curated tactics.
  • Better resumability: coverage and audit history survive context loss.
  • Better wave decisions: unfinished or promising coverage can requeue surfaces intelligently.
  • Better verifier context: verifiers can inspect request audit summaries without reading raw files.

Main Disadvantages And Risks

Risk Why It Matters How To Handle It
Untracked files The repo currently has important untracked files: mcp/lib/*, mcp/redaction.js, and REFACTOR_HARDENING_PLAN.md. Stage them before committing. Otherwise the update will be broken.
More moving parts The architecture is cleaner, but there are more files and imports. Keep module boundaries stable. Do not keep splitting without a concrete reason.
No real MCP smoke test yet Unit tests passed, but the actual stdio/server/client path should be tested once. Run one end-to-end MCP startup and one simple tool call before committing.
Ranking and intel bias Heuristics can over-prioritize noisy surfaces or stale public report patterns. Treat ranking as a suggestion, not truth. Findings still require live proof.
More artifacts to explain Users and agents now need to understand coverage, audit, traffic, and intel files. The README and prompts were updated, but this should be validated in a real run.
Dispatch and transport remain in server This is not perfectly clean architecture. Accept it for now. Splitting transport today is lower value and higher risk.
